# 1.2 用户管理 Demo 设计文档

## 1. 文档定位

本文档用于说明课程大作业中 `1.2 用户管理` 模块的 Demo 设计方案。当前交付目标不是后端完备实现，而是产出一套可演示、可扩展、可被组内其他模块继续复用的前端原型基础。

## 2. 本轮 Demo 目标

本轮 Demo 重点完成以下内容：

- 提供统一的登录、注册、找回密码入口。
- 建立学生、教师、管理员三类角色的界面差异。
- 展示用户列表、用户详情、用户新增/编辑、个人中心等核心页面。
- 体现基于角色的访问控制、菜单隔离和会话冲突提示。
- 为题库管理、考试管理、成绩分析等模块提供统一壳层和用户上下文。

## 3. 角色定位

### 3.1 学生

- 登录后进入个人工作台。
- 只查看和维护自己的资料。
- 不可进入用户管理后台。

### 3.2 教师

- 登录后进入教师工作区。
- 用户管理页面只展示本人及所属学生。
- 教师可以新增学生、维护学生资料、冻结/恢复账号、重置学生密码。
- 教师不能查看管理员信息，也不能承担全局角色配置职责。

### 3.3 管理员

- 登录后进入管理员工作区。
- 可查看全平台教师、学生、管理员账号。
- 负责角色分配、账号状态维护和全局用户管理。
- 管理员与教师在 Demo 中必须显式区分，不能再共用同一职责定位。

## 4. 本轮范围

### 4.1 In Scope

- 登录页
- 注册页
- 找回密码页
- 管理员首页
- 教师首页
- 学生首页
- 用户列表页
- 用户详情页
- 用户新增 / 编辑页
- 个人中心页
- 无权限提示页
- 角色切换
- 会话冲突提示

### 4.2 Out of Scope

- 真实后端接口
- 数据库持久化
- Excel 批量导入解析
- SSO / MFA 等完整认证能力
- 真正的权限策略后台

## 5. 关键设计决策

### 5.1 前端数据驱动

当前 Demo 使用本地存储维护用户、审计日志与会话信息，但页面表现按真实系统交互组织，避免写成纯静态稿。

### 5.2 教师端数据范围收口

为满足需求中“教师只管理自己所属学生”的要求，本轮为学生账号引入 `advisorId` 字段，用于表示负责教师。教师视图下：

- 可以看到自己；
- 可以看到 `advisorId` 等于自己的学生；
- 不能看到管理员；
- 不能看到其他教师；
- 不能编辑自己的角色配置；
- 不能给用户分配管理员角色。

### 5.3 管理员职责独立

管理员视图不再与教师视图混用：

- 首页文案、统计与入口不同；
- 用户管理范围不同；
- 表单中可分配角色；
- 可以指定学生归属教师；
- 可以维护全平台用户状态。

## 6. 页面与路由

| 路由 | 页面 | 访问角色 | 说明 |
| :-- | :-- | :-- | :-- |
| `/login` | 登录页 | 全部 | 统一身份入口 |
| `/register` | 注册页 | 未登录用户 | 学生自注册 |
| `/forgot-password` | 重置密码页 | 未登录用户 | 演示重置流程 |
| `/dashboard` | 首页 | 已登录用户 | 根据角色展示不同首页 |
| `/users` | 用户列表 | Teacher / Admin | 教师看本人及所属学生，管理员看全量 |
| `/users/new` | 新增用户 | Teacher / Admin | 教师新增学生，管理员新增任意角色 |
| `/users/:id` | 用户详情 | Teacher / Admin | 受数据范围控制 |
| `/users/:id/edit` | 编辑用户 | Teacher / Admin | 教师仅可编辑所属学生 |
| `/profile` | 个人中心 | 已登录用户 | 查看和维护本人资料 |
| `/403` | 无权限页 | 已登录用户 | 路由守卫失败提示 |

## 7. 数据模型

### 7.1 User

| 字段 | 说明 |
| :-- | :-- |
| `id` | 用户唯一标识 |
| `loginName` | 登录名 / 学号 / 工号 |
| `name` | 姓名 |
| `roles` | 角色数组 |
| `primaryRole` | 默认角色 |
| `email` | 邮箱 |
| `phone` | 手机号 |
| `classId` | 班级或培养层次 |
| `department` | 学院 / 部门 |
| `advisorId` | 负责教师 ID，仅学生账号使用 |
| `status` | 账号状态 |
| `lastLoginAt` | 最近登录时间 |
| `createdAt` | 创建时间 |
| `updatedAt` | 更新时间 |

### 7.2 Session

| 字段 | 说明 |
| :-- | :-- |
| `sessionId` | 会话 ID |
| `userId` | 当前用户 |
| `loginAt` | 登录时间 |
| `deviceLabel` | 设备标识 |
| `currentRole` | 当前激活角色 |
| `status` | 会话状态 |

### 7.3 Permission

当前 Demo 采用轻量权限编码：

- `user:view`
- `user:create`
- `user:update`
- `user:freeze`
- `role:assign`
- `profile:view`
- `profile:update`

## 8. 演示账号设计

建议演示时重点使用以下账号：

- 管理员：查看全平台账号和角色分配。
- 教师：演示“只看到自己和所属学生”。
- 两个学生账号：演示不同班级 / 培养层次下的学生管理。
- 锁定教师账号：演示状态拦截。
- 多角色账号：演示角色切换能力。

## 9. 演示建议脚本

1. 从登录页进入管理员账号，展示管理员首页和全平台用户列表。
2. 切换到教师账号，展示教师首页与“我的学生”范围收口。
3. 在教师端新增一个学生，说明自动归属当前教师。
4. 在管理员端编辑用户角色，说明管理员与教师职责不同。
5. 使用学生账号登录，展示学生无法访问用户管理后台。
6. 最后演示异地登录冲突提示。

## 10. 与后续模块的衔接

当前用户管理 Demo 已为后续模块预留以下基础：

- 统一的角色上下文；
- 统一的后台壳层；
- 统一的路由守卫；
- 用户范围控制逻辑；
- 账号状态与审计日志基础字段。

后续题库、考试、成绩分析模块可以直接复用这些公共能力继续扩展。
