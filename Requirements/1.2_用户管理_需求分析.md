# 1.2 用户管理需求分析

## 概述
用户管理模块负责系统中所有用户的身份、认证、授权与个人信息管理，支持学生与教师两类主要角色。该模块为考试管理、题库管理、阅卷统计等业务模块提供统一的身份与权限基础设施。

## 目标
- 提供安全、可靠的注册与登录机制。
- 支持基于角色的权限控制（RBAC），便于教师与学生功能隔离。
- 提供会话管理、防多开与并发控制策略以保障考试公平性。
- 提供完善的账号运维功能（重置密码、冻结/解封、用户信息维护）。

## 用户角色
- 学生（Student）：参加考试、查看成绩与个人信息。
- 教师（Teacher）：管理题库、创建考试、查看/导出成绩、管理考生名单。
- 系统管理员（Admin，可选）：系统配置、用户批量导入与权限策略管理。

## 功能需求

### 1. 注册与账号管理
- 功能描述：支持学生/教师注册（邮箱/学号/工号 + 密码），也支持通过管理员后台批量导入账号。
- 触发条件：用户点击注册或管理员上传导入文件。
- 输入字段：用户名、登录名（学号/工号/邮箱）、姓名、角色、密码（或初始密码）、所属班级/部门、联系方式。
- 验证规则：登录名唯一；密码需满足复杂度策略（最小长度 8、包含字母和数字，建议包含特殊字符）；邮箱格式校验。
- 结果：生成用户记录，初始状态可为“未激活”或“已激活”视注册流程而定。
- 异常处理：重复登录名返回明确错误；导入数据含冲突时返回批量导入报告（成功/失败行）。

### 2. 登录与认证
- 功能描述：支持基于会话（Cookie/Server-side session）或基于令牌（JWT）的登录认证，建议后端以 session + httpOnly cookie 为主以提升安全性。
- 登录方式：账号+密码、邮箱验证码（可选）、管理员预置初始密码并首次登录强制修改。
- 多因素认证（可选）：支持短信/邮件验证码二次验证。
- 安全要求：失败登录次数限制与账户锁定策略；密码明文不可存储，使用强哈希算法（如 bcrypt/Argon2）加盐存储。
- 输出：认证成功返回会话标识，前端保存 httpOnly cookie；返回用户角色与权限摘要用于前端 UI 控制。

### 3. 角色与权限管理（RBAC）
- 功能描述：定义并管理系统角色与权限映射，默认角色：Student、Teacher、Admin。
- 权限粒度：资源（考试、题库、成绩等）+ 操作（查看、创建、编辑、删除、导出）。
- 教师角色：具有题库管理、考试创建、阅卷查看、成绩导出权限；部分操作可受更细粒度控制（仅对自己创建的考试有管理权限）。
- 学生角色：只能查看属于自己的考试与成绩，答题时受并发与时间窗限制。
- 管理界面：管理员可新增/编辑角色、分配权限、给用户分配多个角色（如兼任）。

### 4. 会话管理与并发控制
- 功能描述：管理用户会话、支持单一登录/防多开保护（特别针对考试场景）。
- 防多开策略：同一账号在同一考试期间仅允许一个活动会话；检测到第二会话时可提示并阻止或记录并发行为并强制登出早期会话。
- 会话超时：不活动超时设置（如 30 分钟）与考试专属会话（以考试结束时间或答题时长为准）。
- 会话续期：在活动中自动续期；断网重连后允许短时间恢复会话并同步最新答题缓存。

### 5. 账号安全与运维
- 密码重置：通过注册邮箱或管理员重置；重置链接具有效期（如 15 分钟）。
- 账号锁定/冻结：多次错误登录后短暂锁定，或管理员手动冻结（如作弊调查）。
- 日志与审计：记录登录/登出、密码重置、角色变更、关键操作（如用户导入、账号禁用）等审计事件。

### 6. 第三方与 SSO（可选）
- 支持学校统一认证（CAS、OAuth2、SAML）集成，兼容教师与学生的统一账号体系。

## 接口与数据契约（示例）
- POST /api/auth/register
  - 输入：{loginName, password, name, role, classId, email}
  - 输出：{success, userId, message}
- POST /api/auth/login
  - 输入：{loginName, password}
  - 输出：{success, token/sessionCookie, user:{id, name, role, perms}}
- POST /api/users/import
  - 输入：CSV/Excel 文件
  - 输出：{success, reportUrl}
- GET /api/users/{id}
  - 输出：用户详细信息（不包含密码哈希）
- POST /api/auth/reset-password-request
  - 输入：{email}
  - 输出：{success, message}

## 数据模型（核心字段示例）
- User
  - id, login_name, password_hash, salt, name, role(s), email, phone, class_id, status (active/disabled/locked), created_at, updated_at
- Role
  - id, name, description, permissions
- Permission
  - id, resource, action (e.g., exam:create)
- AuditLog
  - id, user_id, action, resource_type, resource_id, ip, user_agent, timestamp, detail

## 非功能性需求
- 安全：密码哈希、传输 TLS、httpOnly + Secure cookie、CSRF 防护、失败登录限流。
- 可用性：支持批量导入（异步任务）、管理员操作的异步反馈与导入报告。
- 扩展性：角色与权限以数据驱动，便于新增角色或细化权限。
- 性能：用户表常用查询需索引（login_name、email、class_id）。
- 合规性：敏感数据（密码、验证码）不在日志记录；保留审计日志至少 90 天（或按学校政策）。

## 异常与边界情况
- 重复注册：返回明确提示并指引找回密码。
- 密码策略未满足：返回具体校验项。
- 批量导入格式错误：提供错误行号与原因。
- 并发登录冲突：在考试期间以防多开策略为准，记录并通知管理员。

## 测试要点
- 注册/登录正反向用例覆盖（含密码强度、邮箱验证）。
- 会话并发：模拟同一账号多窗口登录并验证防护逻辑。
- 角色/权限：验证教师不能越权访问管理员功能，学生不能访问教师管理接口。
- 恢复场景：断网重连后答题缓存与会话恢复。
- 审计日志：关键事件是否完整记录并可导出。
